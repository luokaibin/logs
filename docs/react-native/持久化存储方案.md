# React Native 持久化存储方案

本文档记录在扩展 React Native 支持时，**客户端日志持久化**的技术选型结论与依据，便于后续实现与评审对齐。

---

## 1. 决策结论

| 项目 | 说明 |
|------|------|
| **选用方案** | **react-native-quick-sqlite** |
| **类型** | JSI 版 SQLite，直接走 JavaScript Interface，避免传统 Bridge 序列化开销 |
| **适用范围** | 与当前 Web 端 `LogStore`（IndexedDB）语义对齐的 **SQLite 实现层**（多表、BLOB、批量读写） |

---

## 2. 背景与约束

- 现有浏览器实现将日志、去重摘要、元数据分表存放在 **IndexedDB**（见 `logs/common/LogStore.js`）；React Native 无 IndexedDB，需在原生侧用等价能力承接。
- 期望能力概览：
  - 多「表」：日志主体（二进制 protobuf）、digest 去重、meta KV；
  - 支持 **BLOB**、事务、索引及按条件删除（与当前逻辑一致）；
  - **较高写入与批量读出**性能（定时/阈值 flush）。
- 工程约束：
  - 使用 **JSI** 系实现；
  - 同时可在 **Expo（Development Build）** 与 **裸 React Native** 工程中使用；
  - **不强制**兼容极旧 RN / 旧架构（旧版支持非必须）。

---

## 3. 候选与取舍

| 方案 | 结论 |
|------|------|
| **AsyncStorage / 纯 KV** | 难以自然表达多表 + 索引 + 按时间清理 digest，匹配度低，不作为主存储。 |
| **MMKV** | 适合元数据或大块缓冲；多行日志 + 去重表需自行拼装，不如 SQL 直接。 |
| **文件追加** | 可做日志流，但 digest/meta 仍需侧车存储，维护成本高。 |
| **SQLite（通用）** | 与当前 IndexedDB 分表 + BLOB 模型 **匹配度最高**。 |
| **expo-sqlite** | Expo 官方、新版为 JSI；若全线 Expo 很合适。本仓库需兼顾 **无 Expo 的裸 RN**，不作为唯一绑定。 |
| **op-sqlite** | JSI、性能突出；与 quick 同属中立原生库。生态与资料量相对 **react-native-quick-sqlite** 略少，作为备选即可。 |
| **react-native-sqlite-storage** 等老 Bridge 方案 | 性能与吞吐弱于 JSI 系，新实现不采用。 |

---

## 4. 选择 react-native-quick-sqlite 的理由

1. **JSI**：满足高性能、低延迟批量读写预期。
2. **跨工程形态**：同一依赖可服务于 **裸 RN** 与 **Expo Dev Client**（需自定义开发构建，因含原生代码，与 Expo Go 不兼容属预期行为）。
3. **与现有设计对齐**：可用标准 SQL 复刻 `b_dat` / `digestCache` / `meta` 的职责（BLOB 存单条 `encodeLog` 结果等）。
4. **社区与集成**：在 RN 生态中采用面较广，便于后续问题排查与 ORM/工具链可选集成。

---

## 5. 使用注意（实现阶段）

- **Expo**：须使用 **Development Build / EAS Build**，不能依赖默认 Expo Go。
- **实现时**应在业务层抽象 **存储接口**（与具体驱动解耦），便于单测或将来评估 op-sqlite 等替换。
- 具体表结构、迁移策略在落地 **RN 版 `LogStore`** 时再行细化，并与 Web 端 protobuf 编解码保持一致。
