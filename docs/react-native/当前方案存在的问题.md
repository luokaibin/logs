# 当前方案在 React Native 场景下的问题

本文档面向在 **React Native** 中复用或扩展本仓库（logbeacon）日志采集能力时的现状与障碍，便于后续在 `feature/react-native-support` 分支上有针对性地设计与实现。

---

## 1. 架构假设：浏览器 + Service Worker

- 主流程依赖 **Service Worker**（`browser/beacon.js`、`*/beacon-sw.js`）接收日志、运行聚合与上报。
- React Native **没有 Service Worker**，也无法使用 `navigator.serviceWorker.register` 等 API。
- **结论**：不能在不改架构的情况下「原样」把浏览器端 beacon 搬到 RN；需要 **主线程或其它 RN 可运行环境** 内实现等价的数据通道与聚合逻辑。

---

## 2. 日志分发路径绑定 `window`

- `core/logs.js` 仅在 `typeof window !== "undefined"` 时调用 `sendLog`。
- `common/utils.js` 中 `sendLog` / `sendEvent` 依赖 **`window.dispatchEvent`、`CustomEvent`**，以及向 SW **`postMessage`**。
- React Native 虽可能暴露全局 `window`（兼容层），但 **事件模型与浏览器不一致**，该路径在 RN 上不可靠或不可用。
- **结论**：需要 **平台无关的注入或抽象**（例如显式区分 `web` / `native`，或为 RN 提供单独的 `sendLog` 实现）。

---

## 3. 环境元数据：`getLogExtraInfo()` 强依赖 DOM 与 Web 存储

- `getLogExtraInfo()` 依赖 `window.document`、`location`、`referrer`、`navigator.userAgent`、**`localStorage` / `sessionStorage`**（用于 client UUID、sessionId）。
- 在 RN 中通常 **无 `document` / 无完整 Web 存储语义**；若仅判断失败返回 `{}`，则 **丢失设备与会话维度的关键字段**，且后续仍可能错误走 `sendEvent`。
- **结论**：需要 **React Native 专用元数据**（如 `Platform`、`Dimensions`、应用版本、路由名、可选设备 ID），以及 **AsyncStorage / MMKV 等** 持久化 UUID 与 session。

---

## 4. 持久化：`LogStore` 基于 IndexedDB

- `common/LogStore.js` 使用 **`idb` + IndexedDB**（`beacon-db` 等）。
- React Native **不提供 IndexedDB**。
- **结论**：不能直接复用当前 `LogStore`；需 **存储抽象层** + RN 侧实现（AsyncStorage、MMKV、SQLite 等），并保持与 `LogAggregator` 所需读写语义兼容（日志批次、meta、摘要缓存等）。

---

## 5. 生命周期：页面可见性与卸载

- 浏览器侧使用 **`visibilitychange`、`beforeunload`** 等触发 flush 或上报。
- RN 应使用 **`AppState`**、前后台切换、可选 **`NetInfo`**（离线队列）等；应用被系统杀掉时的行为与 Web 也不相同。
- **结论**：需要 **独立的 RN 生命周期适配**，不能复用现有 `browser/beacon.js` 中的页面事件绑定。

---

## 6. 打包与入口

- `package.json` 的 `exports` 面向 **浏览器 dist**（含 `./sls/beacon`、`./loki/sw` 等），未区分 React Native 入口。
- 若将 SW、IndexedDB 相关代码打入 RN 包，会引入 **无效依赖与潜在运行时错误**。
- **结论**：建议增加 **专用子路径导出**（例如 `logbeacon/rn`）或单独包，并在构建层面 **排除** SW / `idb` 浏览器专用路径。

---

## 7. 网络与安全

- 现有方案使用 `fetch` 上报；RN 一般可用，但需注意：
  - **iOS ATS**、自签名证书、内网地址等配置；
  - 与浏览器一致的 **压缩体、Content-Type、后端鉴权** 需在 RN 侧对齐。
- **结论**：协议层可复用（gzip + 各后端 `logEncoder`），但需在 RN 集成测试中 **验证与现有 beacon 接口完全兼容**。

---

## 8. 可复用部分（对照）

以下模块在抽象存储与分发后，**较适合复用**到 RN：

- 日志级别与拦截思路（`loglevel` + Proxy）；
- `serializeLogContent`、去重与过滤（`LogProcessor`）；
- 批量聚合与上报策略（`LogAggregator` 的核心逻辑）；
- SLS / Loki 的 **`logEncoder` 与后端协议**（若仍对接同一服务）。

---

## 9. 小结

| 类别           | 现状问题概要                                       |
|----------------|----------------------------------------------------|
| 运行时         | 依赖 SW，RN 无对应能力                             |
| API 绑定       | `window` / 自定义事件 / SW 消息，不适配 RN         |
| 存储           | IndexedDB，RN 不可用                             |
| 元数据与会话   | DOM + Web Storage，需替换为 RN 语义               |
| 生命周期       | 页面事件，需改为 AppState 等                      |
| 工程化         | 缺少 RN 专用入口与依赖隔离                       |

后续在本分支上的工作应围绕：**存储抽象、分发抽象、RN 元数据与生命周期、独立入口与构建**，在尽量复用编码与上报协议的前提下完成扩展。
